# 项目文档参考

## 1. 统一约定

- 默认语言：中文
- 目标读者：项目使用者、部署人员、维护人员
- 表达风格：直接、具体、步骤化
- 命令示例：优先使用仓库内真实存在的路径和脚本
- 不确定内容：标记为“待确认”

## 2. README 推荐结构

1. 项目名称和一句话说明
2. 支持的项目类型或适用范围
3. 核心功能
4. 运行环境
5. 数据库配置
6. 启动方式
7. 打包发布
8. 默认账号
9. 测试命令
10. 服务管理
11. 目录说明
12. Windows 服务或其他特殊部署说明

## 3. 安装/部署文档推荐结构

1. 文档说明
2. 运行链路
3. 前置条件
4. 环境准备
5. 依赖安装
6. 数据库配置
7. 手动验证
8. 服务安装
9. 启动与停止
10. 状态检查
11. 日志位置
12. 常见问题
13. 最短执行流程

## 4. FAQ 推荐结构

每个问题都按下面格式写：

````markdown
### 问题标题

现象：
- ...

原因：
- ...

解决方法：
1. ...
2. ...

验证：
- ...
````

## 5. 功能说明文档推荐结构

- 先写“这个功能解决什么问题”
- 再写“怎么用”
- 然后写“输入、输出、限制”
- 最后写“边界情况、权限、依赖关系”

## 6. 一致性检查项

- 术语是否统一
- 服务名是否一致
- 端口是否一致
- 环境变量是否一致
- 路径是否一致
- 命令是否可直接复制执行
- 是否区分了 Windows 和 Linux 的命令

## 7. 写作优先级

如果文档内容不够，优先补这些信息：

1. 用户最常会卡住的地方
2. 真实可执行的命令
3. 验证方式
4. 排错方法
5. 术语说明
